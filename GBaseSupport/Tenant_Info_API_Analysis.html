テナント情報API 分析レポート

SC運営の情報鮮度を保証 - 更新作業ゼロ化・誤情報90%削減

背景: SC（ショッピングセンター）では年間10-20%のテナントが入れ替わり、営業時間も季節・曜日で変動。 情報の二重入力は運用負荷とミスの原因。既存のテナント管理システムからデータを自動取得し、 GBase Supportへリアルタイム同期することで、常に正確な情報を顧客に提供。

SC運営者の情報管理ペインポイント

1. テナント入替の追従

年間10-20%のテナント入れ替え。閉店・新規オープンの情報がGBase Supportに反映されるまでタイムラグが発生し、顧客に誤情報を案内してしまう。

2. 営業時間変更

季節営業（夏季延長等）、曜日別営業時間、年末年始・GW特別営業など、変更パターンが複雑で手動更新が追いつかない。

3. 二重入力の負荷

テナント管理システム、Webサイト、GBase Support、館内サイネージなど複数システムへの同一情報入力で運用コスト増大。

4. 誤情報によるクレーム

「サイトには営業中と書いてあったのに閉まっていた」等のクレームが発生。顧客満足度と信頼性に直結。

実装アプローチ

テナント情報の自動同期には、SC側の既存システムとの連携が必要。 SC毎にシステム環境が異なるため、複数の連携方式をサポートする柔軟なアーキテクチャが重要。

A
汎用API連携
標準REST API + Webhook
  • + SC側で汎用的に実装可能な標準インターフェース
  • + JSON形式でテナント情報をPush/Pull
  • + 変更検知時のWebhook通知で即時反映
  • + 一度実装すれば他SCにも展開容易
  • - SC側でのAPI実装が必要
開発工数: GBase側 2週間 + SC側連携開発
B
CSV/Excelインポート
定期バッチ同期
  • + SC側のシステム改修不要
  • + 既存の運用フローに組み込み可能
  • + テナント管理Excelからの直接取込
  • - リアルタイム性がない（日次/週次更新）
  • - 手動アップロード作業が残る
開発工数: GBase側 1週間
C
個別システム連携
専用コネクタ開発
  • + 特定のテナント管理システムと直接連携
  • + 深いデータ連携が可能
  • - SC毎に異なるシステムのため、個別開発コスト大
  • - 汎用性がない
開発工数: SC毎に2-4週間

Webhook受信エンドポイント

# テナント情報更新Webhook
POST /api/v1/tenants/webhook

# リクエストボディ例
{
  "event_type": "tenant.updated",  // created, updated, deleted
  "timestamp": "2025-01-15T10:30:00+09:00",
  "tenant": {
    "id": "T001",
    "name": "スターバックス コーヒー",
    "name_en": "Starbucks Coffee",
    "floor": "1F",
    "category": "カフェ・喫茶",
    "status": "open",
    "hours": {
      "weekday": "10:00-21:00",
      "weekend": "10:00-22:00",
      "holiday": "10:00-22:00"
    },
    "phone": "03-XXXX-XXXX",
    "closed_days": []
  }
}

バルク同期API

  ]
